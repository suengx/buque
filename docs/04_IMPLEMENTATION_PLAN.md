# Implementation Plan：补雀 BuQue 实施计划

## 1. 实施策略

补雀的实施策略是：

> 先跑通最小可信闭环，再逐步扩大范围；先复制现有规则，再引入 Agent 增强；先形成反馈闭环，再进入预测建议。

---

## 2. 一期目标

一期目标不是做“全自动补货系统”，而是做稳定可用的计划监控预警 Agent。

一期应实现：

- 每日自动读取销量、库存、在途、预测等数据
- 复刻现有 Excel 监控逻辑
- 输出断货、滞销、销量异常、预测偏差、数据异常
- 输出风险等级和触发指标
- 对高风险 SKU 做异常解释
- 输出建议动作和人工确认项
- 记录人工采纳、驳回、部分采纳和修正原因

一期不做：

- 自动改正式预测
- 自动采购
- 自动调拨
- 自动促销或清货
- 模型自改阈值
- 未经确认的高风险动作执行

## 4. 里程碑

| 里程碑 | 周期建议 | 目标 | 验收 |
|---|---:|---|---|
| M1：口径冻结 | 3~5 天 | 字段、规则、输出、验收标准确认 | 评审通过 |
| M2：规则跑通 | 1~2 周 | 监控底表 + 规则引擎 | Excel 一致率 ≥95% |
| M3：日报可用 | 1~2 周 | 日报、清单、单 SKU 卡 | 业务认为可用 |
| M4：反馈闭环 | 1 周+持续 | 人工反馈留痕 | 反馈机制跑通 |
| M5：试运行复盘 | 2~4 周 | 稳定运行并复盘 | 决定是否扩大范围 |

---

## 5. 验收指标

| 指标类别 | 指标名称 | 建议目标 | 是否必须 |
|---|---|---|---|
| 规则一致性 | Excel 规则一致率 | ≥95% | 是 |
| 风险识别有效性 | 红橙灯命中率 | 建议 ≥70% | 是 |
| 风险识别有效性 | 误报率 | 持续下降 | 是 |
| 监控时效 | 日报产出时效 | 按业务约定时间前完成 | 是 |
| 人工提效 | 单日分析耗时下降 | 建议下降 50%+ | 建议 |
| 业务采纳 | 建议采纳率 | 建议 ≥50% | 建议 |
| 系统稳定性 | 定时任务成功率 | ≥99% | 是 |

最核心的验收问题是：

> 系统结果是否能与现有 Excel 逻辑对齐，并且业务认为红橙灯清单有处理价值。

---

## 6. 上线策略

### 推荐策略

- 先跑重点 SKU
- 先跑重点仓
- 先跑重点类目
- 先日报，不急于复杂实时交互
- 先红橙灯提醒，不推送所有黄灯
- 先只读数据，不写回业务系统
- 先人工确认，再逐步扩大自动化程度

### 不推荐策略

- 一期直接全量接所有数据源
- 一期直接重做预测模型
- 一期允许模型自动改规则阈值
- 一期自动创建采购、调拨、促销动作
- 没有反馈机制就做学习型优化

---

## 7. 试运行复盘

试运行期间每周复盘：

| 复盘项 | 问题 |
|---|---|
| 规则一致性 | 哪些规则与 Excel 不一致？原因是什么？ |
| 风险有效性 | 红橙灯是否真的值得处理？ |
| 误报 | 哪些预警被认为无效？ |
| 漏报 | 哪些业务认为有风险但系统没发现？ |
| 解释质量 | Agent 解释是否有证据？ |
| 建议动作 | 建议是否具体、可执行、有责任人？ |
| 反馈质量 | 人工是否愿意填写反馈？标签是否够用？ |
| 数据质量 | 哪些字段最容易缺失或异常？ |

---

## 8. 二期进入条件

满足以下条件后，再进入预测建议模块：

- 一期规则稳定运行
- 人工反馈持续沉淀
- 预测版本与最终确认预测可追溯
- 运营计划结构化程度足够
- 异常销量修正规则清晰
- 数据质量问题可控
- 业务已经接受 Agent 作为建议层
