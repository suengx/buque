# Rules and Outputs：风险规则、解释规则与输出标准

## 1. 设计原则

补雀的规则体系分为三层：

1. **规则参数层**：稳定、可配置、可审计。
2. **异常解释层**：基于证据对异常现象排序解释。
3. **建议动作层**：把解释转化为责任明确的处理建议。

Agent 不直接替代规则参数层。Agent 只消费规则结果、上下文字段和解释选项库，输出解释与建议。

---

## 2. 风险类型

| 风险类型 | 说明 | 常见触发信号 |
|---|---|---|
| 断货风险 | 当前库存无法覆盖未来销售与采购交期 | DOS 低、销量突升、在途延期、重点链接 |
| 滞销风险 | 库存覆盖天数过长或动销持续走弱 | DOS 高、低动销、库龄高、季节尾声 |
| 销量异常 | 近期销量显著偏离正常节奏 | 近 3 天 / 近 15 天 / 近 30 天比值异常 |
| 预测偏差 | 预测与实际销售或运营计划不一致 | 预测误差扩大、运营计划未同步 |
| 数据异常 | 字段缺失、口径冲突、数据过期或映射错误 | 负库存、ETA 缺失、重复 SKU、刷新异常 |

---

## 3. 风险等级

| 等级 | 含义 | 处理方式 |
|---|---|---|
| 绿灯 | 正常 | 进入常规监控 |
| 黄灯 | 轻度风险 | 进入摘要或次级清单，减少噪音 |
| 橙灯 | 中度风险 | 进入日报重点清单，建议责任人确认 |
| 红灯 | 高风险 | 即时提醒，要求当天确认并形成动作结论 |

## 5. 解释规则

解释规则遵循：

> 先识别现象，再选择候选解释；先看直接证据，再看推断证据；数据异常优先于业务判断。

### 典型解释规则

| 异常类型 | 现象 | 主解释 | 次解释 | 建议动作 |
|---|---|---|---|---|
| 销量突升 | 近 7 天销量 / 近 30 天销量 ≥ 1.5 | 运营放量导致 | 促销刺激导致 | 先与运营确认是否为计划内放量 |
| 销量突升 | 销量突升且 DOS 同步走低 | 真实放量叠加断货风险抬升 | 运营放量导致 | 先升级断货预警，再确认放量真实性 |
| 销量突升 | 缺货恢复后 3 天内销量冲高 | 恢复性回补导致 | 数据异常待复核 | 标记恢复性销量，不直接外推长期趋势 |
| 销量下滑 | 近 7 天销量 / 近 30 天销量 ≤ 0.7 | 需求走弱风险 | 运营收量导致 | 提醒运营确认流量、转化、价格变化 |
| 销量下滑 | 销量下滑且可售偏低 / 频繁断货 | 供给受限导致表观销量下降 | 需求走弱风险 | 先排除供给限制，不直接下调需求 |
| 低 DOS | DOS 低于红灯或橙灯阈值 | 真实断货高风险 | 在途延期导致风险抬升 | 计划主管与运营主管确认催交 / 采购 / 控量 |
| 低 DOS + 在途缓释 | DOS 红灯但 ETA 安全窗口内可覆盖 | 短期风险可控，需关注到货兑现 | 真实断货高风险 | 跟踪到货兑现；延期则升回红灯 |
| 高 DOS / 滞销 | DOS 高于滞销阈值 | 去化持续弱 | 计划补货偏多 | 降低后续补货节奏，并同步运营评估去化动作 |
| 预测偏差 | 实际销量与预测偏差持续放大 | 运营计划未同步到预测 | 活动放量未计入 | 要求运营补充计划，再决定是否修正预测 |
| 数据异常 | 关键字段缺失 / 口径不一致 | 销量数据缺失或延迟 | 库存数据口径异常 | 先修复数据，再重新生成判断 |

## 6. 冲突处理原则

| 冲突场景 | 处理原则 |
|---|---|
| 运营计划证据 vs 统计推断 | 有直接计划证据时，优先于统计推断 |
| 供给受限 vs 需求走弱 | 有缺货或低可售时，先排供给侧原因 |
| 恢复性销量 vs 长期需求增长 | 缺货恢复窗口内，不直接外推为长期趋势 |
| 数据异常 vs 业务判断 | 数据异常未排除前，不输出强业务结论 |
| 多个原因都可能成立 | 输出主解释、次解释、第三解释，并说明证据排序 |

## 9. 单 SKU 分析卡模板

```text
日期：2026-04-24
SKU：ABC-001
产品名称：42in Fire Table
风险结论：断货风险 - 红灯

核心触发：
- DOS=12
- 近3天销量较近15天提升42%
- 在途ETA晚于安全窗口

Agent初步判断：
该SKU销量提升更像真实放量，而非短期噪音；若运营计划已确认活动持续，则当前库存覆盖不足。

建议动作：
1. 立即向运营确认未来7~14天销售计划。
2. 若放量属实，评估催交或调拨。
3. 在人工确认前不直接修改正式预测。

责任角色：计划主管
是否必须人工确认：是
建议时效：当天确认
人工反馈位：采纳 / 驳回 / 部分采纳 + 原因
```

## 11. 输出风格

输出必须满足：

- 结论先行
- 指标清楚
- 证据明确
- 建议具体
- 责任角色明确
- 时效明确
- 高风险动作需要人工确认
- 数据异常要先拦截
